# 小说 Wiki 内容规范（CONTENT_SPEC）

> 本文件规定 Tobenot-Story-Wiki 这类「小说/世界观 Wiki」应当**产出什么内容、写到什么深度、用什么语体、如何评级**。
> 标准对标 **维基百科（Wikipedia）** 的条目质量体系与 **动漫人物百科（Fandom / 萌娘百科 风格）** 的人物页结构。
> 它是 [README.md](./README.md)「内容结构与配置指南」的补充：README 定义**结构与字段**，本规范定义**内容、质量与语体**。

---

## 0. 定位

小说 Wiki 不是小说的缩写，而是**把虚构世界外化为可检索、可交叉引用、不互相矛盾、达到百科水准的知识库**。服务于：

1. **追更读者**——忘了某角色是谁、什么能力、上次在哪出现。
2. **新读者**——在不被剧透的前提下了解设定骨架，判断是否入坑。
3. **作者本人**——查设定、防矛盾、管伏笔与时间线。

Wiki 的价值 = **连接密度 × 可信度 × 剧透安全 × 语体专业度**。当前仓库条目多为占位 stub，本规范的核心目标就是**把 stub 抬到 B 级以上**（见 §2）。

---

## 1. 核心原则

1. **单一真源（Canonical）**——每个实体只有一个权威页（`globals/*`）；篇章内只写差异（`localNotes`/`role`），不复制正文。

2. **百科语体（Encyclopedic voice）**——第三人称、客观、克制、可溯源。不夸饰、不第一人称、不「小编体」。详见 §6。

3. **剧透分层**——小说 Wiki 失败的头号原因。卡片层（`summary`/`description`/`role`）绝对无剧透；正文层用 `:::spoiler source=…` 隔离剧情揭示。详见 §5。

4. **可溯源（Verifiability）**——每条非显然断言都能追溯到小说出处。无溯源的断言会退化为粉丝臆测。`:::spoiler source="…"` 同时承载「剧透隔离 + 出处」。

5. **连接优于完备**——稀疏但交叉链接充分 > 详尽但孤立。`related` 必须用 `canonicalId`，让链接可解析、可反向追溯。

6. **身份稳定，细节生长**——`canonicalId` 与核心身份确立后不改；新揭示以追加形式进入剧透块，不删早先悬念。

7. **现实视角优先（Out-of-universe first）**——Lead 段与「创作与设定」节用**现实世界视角**（"罗伯特是《不止于纸上的故事》中的角色"）；设定正文用**世界观内视角**（"罗伯特是银月寺保安"）。这是维基百科处理虚构条目的核心惯例。

## 2. 条目质量评级（对标 Wikipedia 1.0 评估体系）

每个条目应能归入下述等级。**新增/修订条目须达到 B 级以上；现存 stub 须逐步提升。**

| 等级 | 标准 | 现状对照 |
|---|---|---|
| **小作品 Stub** | 仅一两句，无结构，无 Lead。 | ❌ 当前 `globals/characters/robert.md` 等 |
| **初级 Start** | 有基本分节，但关键节缺失或单薄，缺溯源。 | 部分事件/地点条目 |
| **C 级** | 内容较全，但组织/可读性/溯源有缺口，剧透控制不严。 | — |
| **B 级** | 内容完整、结构清晰、溯源充分、剧透合规、语体百科化。**准入线。** | 目标基线 |
| **优良 GA** | 全面、可读、无原创研究、关系网密集、有 Lead 独立成段。 | — |
| **特色 FA** | 权威范例水准，可作为其他条目范本。 | — |

### B 级（准入线）硬指标

- 有**独立成段的 Lead**（见 §3.1），读懂 Lead 即可理解该条目。
- **Infobox/身份卡**字段齐全（见 §4 各类型模板）。
- 至少覆盖该类型**必含章节**的 80%。
- 所有剧情揭示在 `:::spoiler` 内且 `source` 非空。
- `related` 全为有效 canonicalId，无断链。
- `type` 为英文枚举；无 frontmatter 漂移（见 §5.2）。
- 语体百科化（§6），无粉丝夸饰、无第一人称。

## 3. 通用条目骨架（维基百科式）

每个条目按下列顺序组织。**Lead 与 Infobox 是与 stub 的本质区别。**

### 3.1 Lead 段（导言）

条目正文**第一段**，独立成段、不加小标题，须能脱离正文独立成立：

- 用现实视角定义主体：「**罗伯特**是《不止于纸上的故事：银月篇》中的角色，银月寺保安队长，拥有停止时间的异能【苍穹缝隙】。」
- 说明出处作品/篇章、核心身份、为何重要（notability）。
- **无剧透**——只交代到「设定骨架」层面，不剧透命运转折。
- 2–4 句，是卡片/搜索摘要之外读者看到的第一信息。

### 3.2 Infobox / 身份卡

紧随 Lead，以结构化列表呈现硬数据。人物按动漫人物百科风格填全；其他类型见 §4。

### 3.3 正文章节

按类型模板（§4）展开。通用规则：

- 先**设定层**（外貌/性格/能力/机制——安全），后**剧情层**（经历/揭晓——剧透块）。
- 章节用 `##`，子节 `###`；不跳级。
- 能用散文处用散文，枚举仅用于真正并列的条目（如能力清单、出场列表）。

### 3.4 收尾节

- **人际关系/出现于/影响**：连接到其他 canonicalId。
- **出处/外部链接**：作品链接、原始素材出处。

## 4. 各类型条目规范

### 4.1 人物 characters（动漫人物百科式，最高价值）

**Infobox 字段（尽量填全）：**
姓名 / 别名aliases / 所属作品 / 身份职业 / 阵营归属 / 性别 / 年龄 / 生日 / 身高 / 异能 / 武器 / 声优或形象来源 / 首次登场 / 状态。

**标准章节序列：**

1. **Lead**（现实视角，无剧透）
2. **身份卡**（Infobox）
3. **外貌**（安全；建议配图）
4. **性格**（安全）
5. **能力与异能**——每个异能一个子节：名称/机制/限制与代价/觉醒时机（觉醒进剧透块）；可附等级或战力定位。
6. **经历**——按作品/篇章分小节，剧情转折用 `:::spoiler`。
7. **人际关系**——每个相关角色一个 `###` 子节，标明关系类型（敌对/同僚/亲属/师徒/恋人），链向 canonicalId。
8. **名言**（选填）——角色代表性台词，注明出处。
9. **轶事/设定细节**（选填）——名字由来(词源)、创作构思等，现实视角。
10. **出现于**——由 typeIndex 自动聚合，可手写补充。
11. **出处与外部链接**

> Lead + 身份卡 + 外貌 + 性格 + 能力 + 经历，是人物 B 级的最低章节集。

### 4.2 地点 locations（维基百科式）

Lead（现实视角）→ 身份卡（类别/位置/归属/重要性/建立时间）→ 地理与设施（子地点可嵌套）→ 历史 → 居民与势力（链接）→ 战略与剧情意义（spoiler）→ 出处。

### 4.3 事件 events（剧情最密集，剧透最重）

Lead（**只交代背景与性质，不剧透结果**）→ 身份卡（时间/地点/双方/性质）→ 起因 → 经过（全部进剧透块，可按场景分子节）→ 结果与影响（剧透块）→ 关联事件（因果链：破晓行动 → 清扫行动）→ 出处。按 `timeline`/`order` 排序。

### 4.4 物品 items（如能量之体【折射】）

Lead → 身份卡（类别/来源/当前持有者）→ 属性与能力 → 已知持有者（链接）→ 剧情作用（spoiler）→ 出处。

### 4.5 概念 concepts（如异能者/保留地法/灵能盾）

Lead → 定义 → 机制 → 世界内规则/社会影响 → 实例列举 → 世界内历史。概念页是**设定词典**，保持中立无剧透；具体角色觉醒剧情留在角色页，此处仅链接。

## 5. 剧透、溯源与 frontmatter

### 5.1 三层剧透隔离

| 层 | 位置 | 策略 |
|---|---|---|
| **卡片层** | `summary`/`description`/`role` | **绝对无剧透**；`summary` 纯文本无 Markdown。 |
| **身份层** | Lead、Infobox、身份卡 | 默认无剧透；死亡等敏感状态写「未知」+剧透块说明。 |
| **正文层** | 经历/经过/揭晓 | 一切剧情揭示用 `:::spoiler source="出处"` 包裹。 |

剧透块语法：

```markdown
:::spoiler source="《不止于纸上的故事：银月篇》破晓行动"
此处是剧透内容……
:::
```

`source` 必填，写明作品·篇章·（可选）章节/场景。臆测标注 `（推测）` 或 `> 注：`，不得与事实混排。

### 5.2 frontmatter 技术规范（对齐 contentService.js）

> ⚠️ **当前仓库存在漂移，本节为强制标准，迁移时一并修正。**

| 字段 | 要求 | 备注 |
|---|---|---|
| `title` | 必填，中文显示名 | |
| `type` | 必填，**英文枚举** `characters\|locations\|events\|items\|concepts` | ❌ 现有部分文件写成 `"事件"`/`"地点"`/`"概念"`，与路径派生 type 不一致，必须改回英文。 |
| `canonicalId` | 全局权威必填，点分式 `character.robert` | 篇章引用条目也填，指向其全局权威。 |
| `summary` | 纯文本、无 Markdown、无剧透、一句话 | 卡片摘要；优先级 `summary > description`。 |
| `description` | 较长安全描述，无剧透 | 详情页/兜底。 |
| `related` | **canonicalId 列表** | ❌ 现有写成中文名 `["银月寺","罗伯特"]` 无法解析，必须改 canonicalId。 |
| `tags` / `aliases` / `image` | 选填 | `image` 用 `/images/<type>/<slug>.webp`。 |
| `localNotes`/`role`/`firstSeenAt`/`lastSeenAt`/`chapters` | 仅篇章引用条目 | 只写差异，不复制全局正文。 |

## 6. 百科语体（Encyclopedic voice）

把 stub 抬到 B 级，**语体**和内容同样重要。

- **人称与时态**：第三人称。设定用现在时（「罗伯特是银月寺保安」），已发生剧情用过去时。
- **现实视角 vs 世界观内视角**：Lead 与「创作/轶事」节用现实视角（「罗伯特是……中的角色」）；设定正文用世界观内视角。
- **克制不夸饰**：「超强的异能」→「可停止时间的异能」；「帅气地击败」→「在时停中制服」。
- **不第一人称/不读者称呼**：禁「我们」「你」「小编」「相信大家」。
- **可溯源**：非显然断言带 `:::spoiler source=…` 或行内注明出处；无出处不写。
- **中立（NPOV）**：不站队、不吹捧角色；争议性解读标「推测」。
- **散文优先**：能成段叙述处不用纯 bullet；bullet 仅用于并列条目（能力清单、出场表）。

---

## 7. 反模式（禁止）

1. ❌ `summary`/`description`/`role` 含剧透（泄露到卡片）。
2. ❌ `related` 用中文名而非 canonicalId（断链）。
3. ❌ `type` 写中文枚举（破坏 typeIndex）。
4. ❌ 篇章引用条目复制全局正文（应只写差异）。
5. ❌ 把粉丝推测当事实写进正文。
6. ❌ 删除悬念描述以迎合揭晓（应追加剧透块）。
7. ❌ 无 Lead 段、无 Infobox 的「一句话条目」长期留存（须提升至 B 级或标记 Stub 待补）。
8. ❌ 粉丝夸饰、第一人称、bullet 堆砌代替散文。

## 8. 维护与校验清单

新增/修订内容后逐项确认：

- [ ] 有独立成段的 Lead（现实视角、无剧透）。
- [ ] Infobox/身份卡字段齐全；该类型必含章节覆盖 ≥80%。
- [ ] `type` 英文枚举；`canonicalId` 点分式且全局唯一。
- [ ] `related` 全为已存在 canonicalId（无断链）。
- [ ] `summary`/`description`/`role` 无剧透；`summary` 纯文本。
- [ ] 剧情揭示全在 `:::spoiler source=…` 内，source 非空。
- [ ] 篇章引用条目未复制全局正文，仅写差异。
- [ ] 语体百科化（§6），无夸饰/第一人称。
- [ ] 复用实体已建 `globals/*`；仅本篇实体可暂留篇章内。
- [ ] 内容变更后 `npm run build:index` 重建搜索索引（`dev`/`build` 已自动触发）。
- [ ] 条目自评等级 ≥ B；未达标者在 commit 说明中标注 `stub` 待补。

> 建议后续补校验脚本：重复 `canonicalId`、断链 `related`、非法 `type` 枚举、`summary` 含 Markdown/剧透关键词、无 Lead 段检测。README 末尾已留此 TODO。
